## Elite Best Practices (Vercel + Official React)

### Eliminating Waterfalls
- **Parallel data fetching**: Use `Promise.all()` for independent async operations â€” never chain independent fetches sequentially
- **Colocate data fetching**: Fetch data where it's used, not in a parent wrapper that passes it down
- **Preload patterns**: Start fetching before the component that needs data mounts (e.g., hover prefetch on links)
- **Avoid client-side fetch cascades**: If component A fetches, then renders B which fetches, then renders C which fetches â€” that's 3 round trips. Flatten to parallel

### Bundle Size Optimization
- **Import directly from subpaths**: `import { Button } from '@radix-ui/react-button'` not from barrel `index.ts`
- **Tree-shake aggressively**: Named imports only â€” `import { motion } from 'framer-motion'` not `import * as FM`
- **Lazy load below-fold components**: `React.lazy()` + `<Suspense>` for anything not visible on initial paint
- **Analyze bundles**: Use `@next/bundle-analyzer` or `rollup-plugin-visualizer` to find bloat
- **Avoid full lodash imports**: Use `lodash-es` or individual `lodash/get` imports
- **Dynamic imports for heavy libs**: `const Chart = dynamic(() => import('./Chart'), { ssr: false })`
- **Never import an entire icon library** â€” import individual icons: `import { Search } from 'lucide-react'`

### Re-render Optimization
- **Subscribe to derived state**: In Zustand, use selectors `useStore(s => s.count)` not `useStore()` to avoid re-rendering when unrelated state changes
- **Avoid objects as deps**: `useMemo`/`useCallback` deps should be primitives when possible
- **Move state down**: If only one child needs the state, put `useState` there â€” not in the parent
- **Split contexts**: One context per concern â€” `<ThemeContext>` and `<AuthContext>` not `<AppContext>`
- **Stable references**: Wrap event handlers in `useCallback` only when passing to memoized children
- **Never create components inside render**: `function Parent() { function Child() {} }` â€” Child gets remounted every render

### Rendering Performance
- **Virtualize long lists**: Use `@tanstack/react-virtual` or `virtua` for lists >50 items
- **Use `content-visibility: auto`** on hidden/below-fold sections for paint containment
- **Avoid layout thrashing**: Never read DOM measurements (`getBoundingClientRect`, `offsetHeight`) during render
- **Prefer uncontrolled inputs**: Use `defaultValue` + `ref` when you don't need per-keystroke state updates
- **Use `useTransition`** for non-urgent state updates (search filtering, tab switching)
- **Use `useDeferredValue`** for expensive computations derived from fast-changing values

### Server-Side Performance (with Next.js)
- **Use `React.cache()`** to deduplicate data fetches across Server Components in the same request
- **Stream with Suspense**: Wrap slow data components in `<Suspense>` with a fallback for progressive loading
- **Server Components by default**: Only add `'use client'` for interactivity (onClick, useState, useEffect)
- **Pass serializable props**: Server â†’ Client boundary only accepts JSON-serializable data
- **Move client logic to leaves**: Keep `'use client'` components as deep/small as possible

### Client-Side Data Fetching
- **TanStack Query for server state**: Automatic caching, deduplication, background refetch, pagination
- **SWR pattern**: Show stale data instantly, revalidate in background â€” never show loading spinners for cached data
- **Optimistic updates**: Update UI immediately on mutation, roll back on error â€” crucial for perceived performance
- **Prefetch on hover/focus**: `queryClient.prefetchQuery()` when user hovers a link
- **Infinite queries**: Use `useInfiniteQuery` for paginated lists â€” not manual offset tracking

### Advanced Patterns
- **Compound components**: `<Select><Select.Trigger/><Select.Content/></Select>` â€” share state via context internally
- **Render props / children as function**: For maximum flexibility when composition alone isn't enough
- **Polymorphic components**: `<Button as="a" href="/">` â€” use `React.ElementType` for the `as` prop type
- **Form state machines**: Use Zustand or `useReducer` for multi-step forms with validation states
- **URL as state**: Store filters, tabs, pagination in URL params â€” use `nuqs` or `useSearchParams`
- **Portals for overlays**: `createPortal` for modals/tooltips/toasts â€” escape parent overflow/z-index

### Component Architecture
- One component per file â€” filename matches the component name
- Props interface named `{Component}Props` â€” exported for reuse
- Colocate styles, tests, and stories with the component
- Extract custom hooks for any reusable stateful logic (prefix with `use`)
- Separate "container" (data fetching) from "presentational" (pure UI) components
- Use `forwardRef` for any component that wraps a native element

## Anti-Patterns (NEVER do these)

- Don't use `useEffect` for data transformation â€” compute during render or use `useMemo`
- Don't copy props into state â€” derive from props directly
- Don't mutate state â€” always spread/map to new objects/arrays
- Don't create components inside other components â€” they remount every render
- Don't use `index` as key for dynamic lists â€” use stable unique IDs
- Don't fetch data in `useEffect` when a Server Component or TanStack Query would work
- Don't use `any` in TypeScript React code â€” type your props properly
- Don't use `<div onClick>` for interactive elements â€” use `<button>` or `<a>`
- Don't suppress ESLint exhaustive-deps warnings â€” fix the dependency array
- Don't pass entire store objects as props â€” pass only what the child needs
- Don't use `forceUpdate` or mutate `ref.current` to trigger renders
- Don't nest `<form>` elements
- Don't use `dangerouslySetInnerHTML` without sanitization

## Reference (from official docs)

### Core Principles
- Components are the building blocks â€” each component should do one thing
- State should be minimal â€” derive everything you can from existing state
- Lift state up to the nearest common ancestor that needs it
- Use composition over inheritance â€” pass components as children/props
- Keep components pure â€” same inputs should give same outputs

### Hooks Rules
- Only call hooks at the top level (not inside loops, conditions, or nested functions)
- `useState` for simple local state, `useReducer` for complex state logic
- `useEffect` is for synchronizing with external systems, not for transforming data
- `useMemo` and `useCallback` are for optimization â€” don't use them everywhere
- Custom hooks should start with `use` and encapsulate reusable stateful logic
