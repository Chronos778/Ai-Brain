## Elite Best Practices

### Package Management
- **`bun install`**: 10-100x faster than npm ‚Äî use for all install operations
- **`bun add <pkg>`**: Adds dependency and updates lockfile atomically
- **`bun add -d <pkg>`**: Dev dependencies separated properly
- **`bunx`**: Run binaries without global install ‚Äî replaces `npx`
- **`bun.lockb`**: Binary lockfile ‚Äî commit always, smaller + faster than `package-lock.json`
- **Workspace support**: `"workspaces"` in package.json for monorepo management

### Runtime Features
- **`Bun.serve()`**: Ultra-fast HTTP server ‚Äî faster than Express, matches Rust frameworks
- **`Bun.file()`**: Fast file I/O ‚Äî lazy, only handles I/O when consumed
- **`Bun.sql`** / **`bun:sqlite`**: Built-in SQLite ‚Äî no external dependency needed
- **Native `.env`**: Automatic loading of `.env` files ‚Äî no dotenv package needed
- **Native TypeScript**: Run `.ts` files directly ‚Äî zero config, no transpilation step
- **Native JSX**: Run `.tsx` / `.jsx` directly ‚Äî Bun understands JSX natively
- **`Bun.password`**: Built-in bcrypt-compatible password hashing
- **`Bun.CryptoHasher`**: Fast native hashing ‚Äî SHA-256, MD5, etc.

### Testing
- **`bun test`**: Built-in test runner ‚Äî Jest-compatible API
- **`expect()`**: Full Jest matchers available ‚Äî `toEqual`, `toBe`, `toThrow`, etc.
- **`describe` / `it` / `test`**: Standard test structure works out of the box
- **`--watch`**: `bun test --watch` for auto-rerunning on file changes
- **No config needed**: Tests found automatically in `*.test.ts`, `*.spec.ts` files

### Bundling
- **`bun build`**: Built-in bundler ‚Äî alternative to Vite/Rollup for simple builds
- **Tree shaking**: Automatic dead code elimination
- **Minification**: `--minify` flag for production builds
- **Target selection**: `--target=browser` or `--target=node` for output format

### When to Use Bun vs npm
- **Use Bun**: New projects, personal projects, fast iteration, when Node compat isn't an issue
- **Use npm**: CI environments with limited Bun support, team projects where Bun isn't adopted, packages with known Bun issues
- **Don't mix**: Pick one package manager per project ‚Äî never have both `bun.lockb` and `package-lock.json`

## Anti-Patterns (NEVER do these)

- Don't assume 100% Node.js compatibility ‚Äî check Bun docs for gaps
- Don't use `bun.lockb` if team uses npm ‚Äî pick one package manager
- Don't ignore that some npm packages use Node-specific internals
- Don't mix lockfiles ‚Äî either `bun.lockb` OR `package-lock.json`, never both
- Don't use Bun runtime features if the project needs to run on Node.js too
- Don't skip testing Bun-specific code with `bun test` ‚Äî Jest doesn't understand Bun APIs
