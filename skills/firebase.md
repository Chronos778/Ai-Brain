## Elite Best Practices

### Firestore Data Modeling
- **Denormalize for read performance**: Duplicate data across documents — reads are cheap, joins don't exist
- **Subcollections for 1-to-many**: `users/{userId}/posts/{postId}` — scales independently
- **Document size limit**: 1MB max — keep documents lean, move large data to subcollections
- **Collection group queries**: Query across all subcollections with same name — enable in Firestore indexes
- **Timestamps**: Always use `serverTimestamp()` — client clocks aren't reliable
- **Batch writes**: `writeBatch()` for atomic multi-document operations — max 500 ops per batch
- **Transactions**: `runTransaction()` when reads and writes must be atomic
- **Composite indexes**: Create in `firestore.indexes.json` for compound queries — deploy with `firebase deploy --only firestore`

### Authentication
- **Firebase Auth + provider**: Google, GitHub, Email — rarely build custom auth
- **Auth state observer**: `onAuthStateChanged()` — single listener, not per-component
- **ID tokens**: `user.getIdToken()` for authenticated API calls — verify server-side with `admin.auth().verifyIdToken()`
- **Custom claims**: Admin roles, permissions via `admin.auth().setCustomUserClaims()`
- **Session management**: Use ID tokens (stateless) — avoid server sessions with Firebase

### Cloud Functions
- **Cold start optimization**: Keep function bundles small — lazy-load heavy dependencies
- **Regional deployment**: `functions.region('us-central1')` — closest to your users/DB
- **Idempotency**: Firestore triggers can fire multiple times — use document ID as idempotency key
- **Timeout configuration**: Default 60s — increase for heavy processing, decrease for simple operations
- **Memory allocation**: Default 256MB — increase for image/ML processing
- **Callable functions**: `functions.https.onCall()` — auto-handles auth, CORS, serialization
- **Background triggers**: `onDocumentCreated`, `onDocumentUpdated` — react to Firestore changes
- **Scheduled functions**: `functions.pubsub.schedule('every 24 hours')` for cron jobs
- **Secret management**: `defineSecret()` for API keys — not environment variables

### Security Rules
```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Authenticated users can read their own data
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
    // Public read, authenticated write
    match /issues/{issueId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth != null && 
        (request.auth.uid == resource.data.userId || request.auth.token.admin == true);
    }
  }
}
```
- **Default deny**: Start with no access, open specifically
- **Auth checks**: `request.auth != null` for authenticated-only operations
- **Owner checks**: `request.auth.uid == resource.data.userId` for user-owned data
- **Data validation in rules**: `request.resource.data.title is string && request.resource.data.title.size() < 200`
- **Admin via custom claims**: `request.auth.token.admin == true` — not a field in the document

### Firebase Storage
- **Security rules**: Only authenticated uploads — validate file type and size in rules
- **Structured paths**: `users/{userId}/uploads/{filename}` — organized, easy to secure
- **Resize on upload**: Cloud Function trigger to generate thumbnails automatically
- **Download URLs**: `getDownloadURL()` for public URLs with token — or use signed URLs for temporary access
- **CORS configuration**: Required for web uploads — configure in `cors.json`

### Google Cloud AI Integration
- **Vision API**: Label detection, OCR, face detection, safe search — call from Cloud Functions (not client)
- **Gemini AI**: Text generation, image understanding — use for description generation, content analysis
- **API key security**: Never expose Google Cloud API keys in client code — proxy through Cloud Functions
- **Rate limiting**: Implement quotas — Google APIs have per-project limits
- **Error handling**: Google API errors have specific codes — handle `RESOURCE_EXHAUSTED`, `INVALID_ARGUMENT` specifically

### Deployment
- **Firebase CLI**: `firebase deploy` for all services
- **Selective deploy**: `firebase deploy --only hosting`, `--only functions`, `--only firestore`
- **Preview channels**: `firebase hosting:channel:deploy preview` for staging
- **CI/CD**: GitHub Actions with `firebase-tools` for automated deployments
- **Environment config**: `firebase functions:config:set` for production secrets

### Real-Time Features
- **`onSnapshot()`**: Real-time listeners for live data — dashboard updates, chat, notifications
- **Debounce listeners**: Don't attach per-component — centralize in a service/store
- **Unsubscribe on unmount**: Store unsubscribe function from `onSnapshot()` — call in cleanup
- **Offline persistence**: Firestore has built-in offline support — enable for mobile

## Anti-Patterns (NEVER do these)

- Don't expose API keys in client code — proxy through Cloud Functions
- Don't skip security rules — "test mode" is for development only
- Don't use Firestore as a relational database — denormalize, don't join
- Don't read entire collections — always query with filters and limits
- Don't store large files in Firestore documents — use Firebase Storage
- Don't trust client-side validation alone — validate in security rules AND Cloud Functions
- Don't deploy to production without reviewing security rules
- Don't use `firebase.firestore()` (v8 syntax) — use modular v9+ imports
- Don't put heavy initialization in global scope of Cloud Functions — lazy load
- Don't forget to unsubscribe from `onSnapshot()` listeners

## Reference (from official docs)

### Core Architecture
- Firebase is Google Cloud's app platform — Authentication, Database, Storage, Hosting, Functions
- Firestore: NoSQL document database — collections → documents → fields
- Cloud Functions: Serverless Node.js — triggered by HTTP, Firestore events, Auth events, schedules
- Firebase Hosting: CDN-backed static hosting with SSL — preview channels for staging
- Firebase Storage: Object storage backed by Google Cloud Storage — for user uploads
