## Elite Best Practices

### Type-First Development
- Define types/interfaces BEFORE implementation — they serve as documentation
- Export all prop types: `export interface ButtonProps { ... }` — consumers need them
- Use `z.infer<typeof Schema>` for shared types between client/server (single source of truth)
- Colocate types with their modules — `types.ts` in each feature folder, not one giant types file

### React + TypeScript Patterns
- **Component props**: `interface Props` + destructure in function signature
- **Event handlers**: `React.MouseEvent<HTMLButtonElement>` not generic `any`
- **Ref typing**: `React.RefObject<HTMLDivElement>` / `useRef<HTMLDivElement>(null)`
- **Children**: Use `React.ReactNode` (most permissive) or `React.ReactElement` (strict)
- **Polymorphic as prop**: `<T extends React.ElementType>` with `React.ComponentPropsWithRef<T>`
- **Generic components**: `function List<T extends { id: string }>(props: { items: T[] })`
- **Hook return types**: Let inference work for simple hooks; annotate complex/public ones
- **Context typing**: `createContext<ContextType | null>(null)` + throw in consumer if null

### Advanced Type Patterns
- **Discriminated unions for state machines**:
  ```
  type State = { status: 'idle' } | { status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: Error }
  ```
- **Branded types for IDs**: `type UserId = string & { __brand: 'UserId' }` — prevents mixing user IDs with post IDs
- **Template literal types**: `type EventName = \`on${Capitalize<string>}\`` — for naming pattern enforcement
- **Mapped types for API responses**: `type APIResponse<T> = { data: T; meta: PaginationMeta }`
- **Conditional types**: `type IsArray<T> = T extends any[] ? true : false`
- **Extract/Exclude**: Filter union members — `type SuccessState = Extract<State, { status: 'success' }>`
- **`NoInfer<T>`**: Prevent TypeScript from inferring a type parameter from a specific argument
- **`satisfies` for config**: `const config = { ... } satisfies Config` — validates shape while preserving literal types

### Utility Types Mastery
- `Partial<T>` — all properties optional
- `Required<T>` — all properties required
- `Readonly<T>` — immutable version
- `Pick<T, K>` / `Omit<T, K>` — select/exclude properties
- `Record<K, V>` — object with known key type
- `ReturnType<T>` — extract return type of a function
- `Parameters<T>` — extract parameter types as tuple
- `Awaited<T>` — unwrap Promise types
- `NonNullable<T>` — remove null and undefined
- `ConstructorParameters<T>` — extract constructor parameter types

### Strict Patterns
- **Exhaustive switches**: Always handle `default: never` to catch missing cases at compile time
- **Const assertions**: `as const` for arrays and objects that shouldn't change
- **Readonly collections**: `ReadonlyArray<T>`, `ReadonlyMap<K,V>`, `ReadonlySet<T>` for data that flows down
- **Strict function types**: Always type callback parameters — `(callback: (item: T) => void)` not `(callback: Function)`
- **No implicit returns**: Every code path should explicitly return — enable `noImplicitReturns`

### Error Handling in TypeScript
- **Result pattern**: `type Result<T, E = Error> = { ok: true; value: T } | { ok: false; error: E }`
- **Type-safe error boundaries**: Typed error classes extending `Error` with additional metadata
- **Zod for runtime validation**: Parse external data (API responses, form data, env vars) at the boundary
- **Never trust external data**: `JSON.parse()` returns `any` — always validate with Zod or a type guard

## Anti-Patterns (NEVER do these)

- Don't use `any` — use `unknown` and narrow with type guards
- Don't use `enum` — use `as const` objects or union types
- Don't use `!` (non-null assertion) — handle null/undefined properly
- Don't overtype — if inference works, omit the annotation
- Don't use `@ts-ignore` — use `@ts-expect-error` with explanation
- Don't use `Function` type — use specific signatures `() => void`
- Don't use `Object` or `{}` as types — use `Record<string, unknown>` or specific interfaces
- Don't export namespace/module — use ES module exports
- Don't type everything as `string` when a union of literals is more precise
- Don't cast with `as` when a type guard or narrowing check would work

## Reference (from official docs)

### Core Principles
- Use strict mode (`strict: true`) — it catches more bugs at compile time
- Prefer `interface` for object shapes that may be extended, `type` for unions/intersections
- Use `unknown` over `any` — it forces you to narrow the type before using it
- Leverage type inference — don't annotate what TypeScript can figure out
