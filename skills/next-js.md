## Elite Best Practices (Vercel + Official Docs)

### App Router Architecture
- **Server Components by default**: Every file in `app/` is a Server Component unless marked `'use client'`
- **Push `'use client'` to leaf components**: Don't make a whole page client — extract just the interactive part
- **Layout composition**: `layout.tsx` wraps route segments — use for nav, sidebar, providers
- **Loading UI per segment**: `loading.tsx` creates instant loading states per route
- **Error isolation**: `error.tsx` catches errors per segment without breaking the whole app
- **Not found**: `not-found.tsx` per route segment for granular 404 pages
- **Route groups**: `(marketing)/` and `(dashboard)/` — different layouts, same URL structure

### Data Fetching (Zero-Waterfall)
- **Fetch in Server Components**: Direct `async/await` — no `useEffect`, no client-side loading states
- **Use `React.cache()`**: Deduplicate identical fetches across components in the same request
- **Parallel fetching**: `const [posts, user] = await Promise.all([fetchPosts(), fetchUser()])`
- **Stream with Suspense**: Wrap slow components in `<Suspense fallback={<Skeleton/>}>` for progressive rendering
- **Revalidation strategies**:
  - Time-based: `fetch(url, { next: { revalidate: 3600 } })` — revalidate every hour
  - On-demand: `revalidatePath('/posts')` or `revalidateTag('posts')` in Server Actions
- **Server Actions for mutations**: `'use server'` functions replace API routes for form submissions
- **Don't fetch in layouts**: Layout data should be layout-specific (nav, auth) — not child data

### Routing Patterns
- **File-based routing**: `app/blog/[slug]/page.tsx` → `/blog/my-post`
- **Catch-all routes**: `[...slug]/page.tsx` for flexible hierarchies
- **Parallel routes**: `@modal/` slots for concurrent page rendering (modal + background)
- **Intercepting routes**: `(.)photo/[id]` for modal → full page navigation patterns
- **Dynamic params**: `generateStaticParams()` for pre-rendering dynamic routes at build time
- **Route handlers**: `app/api/route.ts` for webhooks, third-party integrations — NOT for data fetching in your own app

### Performance Optimization
- **`next/image`**: Always use for images — automatic WebP/AVIF, lazy loading, blur placeholder
- **`next/font`**: Zero layout shift — `const inter = Inter({ subsets: ['latin'] })` in layout
- **`next/link`**: Automatic prefetching of linked pages on viewport intersection
- **Dynamic imports**: `dynamic(() => import('./HeavyChart'), { ssr: false })` for client-only heavy components
- **Bundle analysis**: `@next/bundle-analyzer` to find client-side bloat
- **Metadata API**: `generateMetadata()` for dynamic SEO per page — title, description, OG images
- **Preconnect to external origins**: `<link rel="preconnect" href="https://fonts.googleapis.com" />`
- **ISR (Incremental Static Regeneration)**: Use for content that updates occasionally — serve static, rebuild in background

### Middleware Patterns
- **Edge middleware**: `middleware.ts` at project root for auth checks, redirects, geolocation
- **Match specific paths**: Use `config.matcher` to avoid running middleware on static assets
- **Keep middleware light**: No database calls — only JWT verification, header manipulation, redirects
- **Geolocation**: `request.geo` for country-based redirects or content

### Server Actions
- **Form mutations**: `<form action={createPost}>` — progressive enhancement, works without JS
- **Optimistic updates**: `useOptimistic()` for instant UI feedback during server mutation
- **Revalidation after mutation**: Call `revalidatePath()` or `revalidateTag()` inside the action
- **Error handling**: Use `try/catch` in Server Actions, return error states to the form
- **File conventions**: Separate server action files with `'use server'` directive at the top

## Anti-Patterns (NEVER do these)

- Don't use `useEffect` for data fetching — use Server Components
- Don't put database/API secrets in `'use client'` components
- Don't create API routes just to fetch data in your own app — use Server Components
- Don't use `getServerSideProps` / `getStaticProps` — those are Pages Router legacy
- Don't make entire pages `'use client'` — extract only the interactive parts
- Don't fetch data in `layout.tsx` that's specific to a child route
- Don't use `router.push()` for simple navigation — use `<Link>`
- Don't disable ISR/caching without reason — Next.js defaults are good
- Don't put heavy libraries in Server Components (they don't benefit from tree-shaking the same way)
- Don't use `window` or `document` in Server Components — they don't exist on the server
