## Elite Best Practices

### Configuration
- **TypeScript config**: Always `vite.config.ts` ‚Äî type-safe, autocompletion
- **Path aliases**: Mirror `tsconfig.json` paths ‚Äî `resolve.alias: { '@': path.resolve(__dirname, 'src') }`
- **Environment variables**: `.env`, `.env.local`, `.env.production` ‚Äî only `VITE_` vars exposed to client
- **Define constants**: `define: { __APP_VERSION__: JSON.stringify(version) }` for build-time constants
- **CSS configuration**: PostCSS + Tailwind in `postcss.config.js` ‚Äî Vite handles it natively

### Build Optimization
- **Code splitting**: Dynamic `import()` creates separate chunks automatically
- **Manual chunks**: `build.rollupOptions.output.manualChunks` for vendor splitting
  ```
  manualChunks: { vendor: ['react', 'react-dom'], ui: ['framer-motion', '@radix-ui/react-dialog'] }
  ```
- **Tree shaking**: ES modules only ‚Äî Rollup removes unused exports automatically
- **CSS code splitting**: Each async chunk gets its own CSS file
- **Asset inlining**: Files < 4KB auto-inlined as base64 ‚Äî configure with `assetsInlineLimit`
- **Dependency pre-bundling**: Vite pre-bundles deps with esbuild ‚Äî customize with `optimizeDeps`

### Dev Server
- **Proxy configuration**: Forward API calls during development
  ```
  server: { proxy: { '/api': { target: 'http://localhost:3001', changeOrigin: true } } }
  ```
- **HTTPS**: `server.https: true` for PWA development or service worker testing
- **Open browser**: `server.open: true` to auto-launch browser on dev start
- **HMR**: Works automatically for React with `@vitejs/plugin-react` ‚Äî instant updates

### Plugin Ecosystem
- **`@vitejs/plugin-react`**: Fast Refresh + JSX transform ‚Äî required for React
- **`vite-plugin-svgr`**: Import SVGs as React components
- **`vite-tsconfig-paths`**: Auto-resolve TS path aliases without manual alias config
- **`rollup-plugin-visualizer`**: Bundle analysis ‚Äî find what's bloating your build
- **`vite-plugin-pwa`**: Service worker + manifest generation for PWAs
- **`@vitejs/plugin-legacy`**: Generate builds for older browsers when needed

### Performance Patterns
- **Dynamic imports for routes**: `const Dashboard = lazy(() => import('./pages/Dashboard'))`
- **Import.meta.glob for bulk imports**: `const modules = import.meta.glob('./modules/*.ts')`
- **Worker imports**: `new Worker(new URL('./worker.ts', import.meta.url))` ‚Äî Vite handles bundling
- **Conditional imports**: `import.meta.env.DEV && import('./dev-tools')` ‚Äî tree-shaken in production
- **Preload directives**: Vite auto-generates `<link rel="modulepreload">` for entry chunks

### Environment Variables
- **Naming convention**: `VITE_API_URL`, `VITE_FIREBASE_KEY` ‚Äî must start with `VITE_`
- **Type-safe env**: Create `env.d.ts` with `ImportMetaEnv` interface for autocompletion
- **Validation**: Parse `import.meta.env` through Zod at app initialization ‚Äî fail fast on missing vars
- **Never `process.env`**: Use `import.meta.env` in Vite projects ‚Äî `process.env` doesn't exist in browser

## Anti-Patterns (NEVER do these)

- Don't use `process.env` ‚Äî use `import.meta.env`
- Don't use CommonJS (`require()`) ‚Äî use ES modules
- Don't put secrets without `VITE_` prefix (they won't be exposed ‚Äî but also won't work)
- Don't put actual secrets WITH `VITE_` prefix ‚Äî they're embedded in the client bundle
- Don't install Create React App ‚Äî Vite is the standard
- Don't skip the visualizer after adding dependencies ‚Äî know your bundle size
- Don't use `resolve.alias` without matching `tsconfig.json` `paths`
- Don't forget `.env.local` in `.gitignore`
